## Description

Production Active Directory environments accumulate stale accounts that become security liabilities - every dormant account is a credential an attacker can leverage without triggering recent-login monitoring. You will build a production-quality automated remediation pipeline in PowerShell, not just a query script.

### Part 1 - Remediation Script (60 pts)

Build `hw-03-remediate.ps1` that does all of the following. The script must declare these parameters, with these exact names and defaults, instead of hard-coding the equivalent values as internal variables - the autograder calls your script with these parameters, so it can't pass them if they don't exist:

```powershell
param(
    [string[]]$ExcludeList = @('svc-backup', 'svc-sql'),
    [int]$StaleDays = 90,
    [int]$PasswordDays = 180,
    [string]$OutputPath = ".",
    [string]$LogPath = ".\remediation-log.txt",
    [switch]$WhatIf
)
```

1. **Discovery** - Query AD for all enabled user accounts where `LastLogonDate` is more than `$StaleDays` days ago **or** where `PasswordLastSet` is more than `$PasswordDays` days ago. Exclude service accounts (accounts whose `Description` contains "Service Account" or whose `SamAccountName` is in `$ExcludeList`).

2. **Export** - Write findings to `$OutputPath\stale-accounts-YYYY-MM-DD.csv` with columns: `SamAccountName`, `DisplayName`, `LastLogonDate`, `PasswordLastSet`, `OU`, `Email`, `Reason` (stale/expired-password/both), `Action` (Disabled/Skipped).

3. **Remediation** - For each non-excluded stale account:
   - Disable the account
   - Set the `Description` attribute to `[DISABLED] Automated policy - stale >$StaleDaysd. Disabled by: $env:USERNAME on $(Get-Date -Format 'yyyy-MM-dd')` (interpolate the actual `$StaleDays` value, not the literal text)
   - Move the account to a designated `OU=Disabled,OU=Users,DC=lab,DC=local` OU (create the OU if it does not exist)
   - Add an entry to `$LogPath` with timestamp, account, and action taken

4. **Error handling** - All AD operations must be wrapped in `try/catch`. If any single account fails, log the error and continue to the next - the script must not crash mid-run. At the end, print a summary: `X accounts disabled, Y skipped (excluded), Z errors`.

5. **Dry-run mode** - `-WhatIf` here is a plain switch parameter, not PowerShell's built-in `SupportsShouldProcess` mechanism (that only auto-generates a `-WhatIf` parameter for functions declared with `[CmdletBinding(SupportsShouldProcess=$true)]`, and its prompts won't give you the custom `[DRY RUN]` text this assignment requires). Write explicit `if ($WhatIf) { ... } else { ... }` branching around every AD-modifying call. When `-WhatIf` is passed, the script prints what it *would* do without making any changes, and every action line shows a `[DRY RUN]` prefix.

6. **CI compatibility** - Guard your `Import-Module ActiveDirectory` call, e.g. `Import-Module ActiveDirectory -ErrorAction SilentlyContinue`. Your script is graded by a CI pipeline that supplies a stand-in module exposing the same cmdlet names (`Get-ADUser`, `Disable-ADAccount`, etc.) without real RSAT installed. An unconditional `Import-Module ActiveDirectory` throws on that runner and fails your build before any of your logic runs.

### Part 2 - Pester Tests (25 pts)

Write `hw-03-remediate.Tests.ps1` using **Pester v5** that includes at least **5 test cases**. Mock every AD cmdlet your script calls (`Get-ADUser`, `Disable-ADAccount`, `Set-ADUser`, `Move-ADObject`, `Get-ADOrganizationalUnit`/`New-ADOrganizationalUnit`) - you never need a live AD connection to test this script. Pass `-OutputPath $TestDrive` and `-LogPath "$TestDrive\remediation-log.txt"` so your tests read/write to Pester's isolated temp directory instead of your real working directory. At minimum:

- Verify the script accepts a `-WhatIf` switch without error, and that `Disable-ADAccount`, `Set-ADUser`, and `Move-ADObject` are never called when it is passed
- Mock `Get-ADUser` to return a test dataset and verify the CSV export contains the correct columns
- Verify excluded accounts (passed via `-ExcludeList`) are not disabled (mock `Disable-ADAccount` and assert with `Should -Invoke Disable-ADAccount -Times 0 -ParameterFilter { ... }` that it is NOT called for excluded users)
- Verify the `Description` field is set with the correct format on a mock account
- Verify the script produces a non-empty log file at `-LogPath` after a successful run

### Part 3 - Write-Up (15 pts)

In your PR write-up (`homework/hw-03.md`), answer:

1. What is the security risk of dormant accounts beyond just "wasted licenses"? Cite one real-world breach where stale credentials were exploited.
2. Your script moves disabled accounts to a separate OU rather than deleting them immediately. What is the operational justification for this design? How long should accounts remain there before deletion, and who should approve the final deletion?
3. Identify one limitation of using `LastLogonDate` as the sole staleness indicator and propose a more reliable alternative or supplement.

## Tips

{: .tip }
`LastLogonDate` and `PasswordLastSet` are read-only *computed* properties - you can't set them directly. `LastLogonDate` is derived from the `lastLogonTimestamp` attribute (not `lastLogon`, which is per-domain-controller, never replicated, and does not drive `LastLogonDate`). Backdate it with `Set-ADUser <user> -Replace @{lastLogonTimestamp = (Get-Date).AddDays(-120).ToFileTime()}`. Backdate `PasswordLastSet` the same way via the underlying `pwdLastSet` attribute: `Set-ADUser <user> -Replace @{pwdLastSet = (Get-Date).AddDays(-200).ToFileTime()}`.

{: .tip }
Use the provided [hw-03-create-test-users.ps1]({{ site.baseurl }}/homework/assets/hw-03-create-test-users.ps1) script to create a ready-made set of test accounts (stale login, expired password, both, an excluded service account, and a clean control account) in your lab AD instead of doing this by hand.

{: .tip }
Pester mocking uses `Mock Get-ADUser { ... }` inside a `Describe` block. You do not need a live AD for tests - that's the point.

## Automated Grading

When you open your PR, a GitHub Actions workflow automatically runs `hw-03-remediate.ps1` against a simulated AD environment (a stand-in module with the same cmdlet names as the real `ActiveDirectory` module, seeded with a known set of fake accounts) and checks the same kinds of behaviors your own Pester tests should cover: discovery filtering, exclusion handling, CSV output, `-WhatIf` no-op behavior, and logging. This is why the parameter names in Part 1 (`-ExcludeList`, `-StaleDays`, `-PasswordDays`, `-OutputPath`, `-LogPath`) must match exactly - the grader calls your script with these parameters by name. A failing automated check does not by itself fail the assignment, but it's a strong signal something is broken before the TA ever looks at it manually. It does not replace Part 2 - your own Pester suite is graded separately for quality, coverage, and correct mocking technique.

### Part 4 - SIEM Integration & Account Lifecycle Policy (30 pts)

**SIEM Shipping Script (15 pts)**

Extend your remediation pipeline with `hw-03-siem-ship.ps1`. This script reads `remediation-log.txt` and ships each entry as a structured JSON event to a Splunk HTTP Event Collector (HEC) endpoint or, if you don't have Splunk, writes to a local file simulating the HEC payload. Each event must include:

- `time` (Unix epoch), `host` (env var), `source` (script name), `sourcetype` (`ad:remediation`)
- `event` object with: `action`, `sam_account_name`, `ou_before`, `ou_after`, `operator`, `reason`, `error` (null if none)

Mock the HEC endpoint in your Pester tests using `Mock Invoke-RestMethod`. Verify the payload schema matches the above structure.

**Account Lifecycle Policy Document (15 pts)**

Write a formal **Account Lifecycle Policy** (`hw-03-lifecycle-policy.md`) covering the full joiner-mover-leaver (JML) process for your organization. The policy must include:

1. **Joiner** - provisioning SLA (how quickly an account must be created after HR onboarding), required approvals, default group assignments, and initial password policy
2. **Mover** - process when an employee changes roles or departments, including access review requirements and timeline for removing old permissions
3. **Leaver** - immediate vs. scheduled disable/delete triggers, account preservation requirements for legal hold, and who has authority to approve early deletion from the `OU=Disabled` OU
4. **Stale Account Definition** - your organization's formal definition (align with your script's 90/180-day thresholds and justify those numbers with a reference to NIST SP 800-53 AC-2 or equivalent)
5. **Exceptions** - process for service accounts, shared accounts, and executive accounts that cannot follow standard JML

The policy should be written as a real organizational document (not a bullet list summary) and be suitable for inclusion in an ISMS policy library.
